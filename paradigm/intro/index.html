    Paradigm

    Paradigm shift

    A person can change their paradigm, either slowly or all at once through an epiphany. You usually need a crisis to bring about an epiphany, so we prefer the first method. (1)

    1. If you‚Äôre used to object-oriented programming (OOP), then you‚Äôll be entering a new paradigm. FRP will appear a bit strange. We can claim all day that FRP is a simple idea, and it is. But simple and easy to understand aren‚Äôt the same thing. The reality is that until your thinking slots into place, you will encounter some challenges.

      FRP rests on certain notions about what‚Äôs important in programming that may go against your current understanding. Without these ideas, FRP is just a way of taking something that should be straightforward and doing it in an eccentric, limiting way. The claimed benefits will be remote.

      We‚Äôre talking about standard functional programming ideas. If you‚Äôve done functional programming, it will be easier to learn FRP. If not, that‚Äôs no problem. You don‚Äôt need to know everything about functional programming to use FRP, and we‚Äôll teach you what you need to know.

    Thinking declaratively: what the program is, not what it does

    In FRP we talk about working in the problem space rather than working in the machine space. Decades of software development have made the authors lazy. We don‚Äôt want to add sequence information to our code if we don‚Äôt have to. We‚Äôll only end up having to debug it.

    The sequence can be derived from dependencies, so you can write less code by leaving the sequence out altogether. You end up with a lot more ‚Äúwhat‚Äù and a lot less ‚Äúhow.‚Äù

    This style is referred to as declarative programming: you tell the machine what the program is, not what it does. You directly describe things and the relationships between them.

    can i console.log all web api exposed in browser

    You can‚Äôt directly console.log all Web APIs exposed by the browser in one command, because they are part of the global environment (window object) and span hundreds of interfaces. However, you can inspect or iterate through the window object to see what‚Äôs available. Here's how you can explore it:

    üîç View all properties of the window object (approximate list of Web APIs):

    console.log(Object.getOwnPropertyNames(window));

    This will log all enumerable and non-enumerable properties on the global window object ‚Äî which includes a huge number of Web APIs (like fetch, localStorage, WebSocket, IntersectionObserver, etc.).

    üß† Filter only functions (API-like):

  if (typeof window[key] === "function") {

  }

    üß™ Example: List all global classes (constructor functions)

  try {

    console.log(Object.getOwnPropertyNames(window)); can i see if formData is present in this

    Yes, FormData is a global Web API, and it will appear in the output of:

    console.log(Object.getOwnPropertyNames(window));

    However, the list is very long, so it can be hard to spot manually. To check specifically for FormData, you can do something like this:

    ‚úÖ Check if FormData exists:

  console.log("FormData is available");

  console.log("FormData is NOT available");
}

    üîç Or filter from the list:

    üß™ Confirm it‚Äôs a constructor (function):

    filter out all DOM-related only

    To filter out only DOM-related APIs from window, you can look for constructors and objects that are part of the DOM specification.

    ‚úÖ Here's a basic way to filter DOM-related APIs:

    const domApis = Object.getOwnPropertyNames(window).filter((name) => {
  try {
    const val = window[name];
    return (
      typeof val === "function" &&
      // Check if it's a class or function related to DOM
      (val.prototype instanceof Node ||
        val.prototype instanceof EventTarget ||
        val.prototype instanceof HTMLElement)
    );
  } catch (e) {
    // Ignore properties that throw on access
    return false;
  }
});

console.log(domApis);

    üí° What this does:

    • It checks if the global property is a function (constructor).

    • Then it tests if it inherits from Node, EventTarget, or HTMLElement, which are base classes for DOM elements.

    • This captures DOM interfaces like HTMLDivElement, Document, Element, Event, etc.
